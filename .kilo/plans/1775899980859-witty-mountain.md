# Plan: Complete order_by Support in GraphQL

## Goal

Add per-column, type-safe `order_by` support to the GraphQL list resolvers with 6 sort direction options. The current `order_by: str | None` parameter on `list_resolver` (line 177) is unused — replace it with a dynamically generated, per-table Strawberry input type.

## Target GraphQL API

```graphql
enum SortDirection {
  ASC
  ASC_NULLS_FIRST
  ASC_NULLS_LAST
  DESC
  DESC_NULLS_FIRST
  DESC_NULLS_LAST
}

# Dynamically generated per table (e.g. for a "users" table):
input UsersOrderBy {
  id: SortDirection
  name: SortDirection
  email: SortDirection
  created_at: SortDirection
  # ... one field per column
}

query {
  users(order_by: UsersOrderBy, limit: 10) {
    nodes { id name email }
    total_count
  }
}
```

## Implementation Steps

### 1. Add `SortDirection` Strawberry enum to `models.py`

Add a `@strawberry.enum` class with 6 members:

```python
@strawberry.enum
class SortDirection(Enum):
    ASC = "ASC"
    ASC_NULLS_FIRST = "ASC_NULLS_FIRST"
    ASC_NULLS_LAST = "ASC_NULLS_LAST"
    DESC = "DESC"
    DESC_NULLS_FIRST = "DESC_NULLS_FIRST"
    DESC_NULLS_LAST = "DESC_NULLS_LAST"
```

Place in `models.py` alongside `ResolverType` since it's a shared enum used across the GraphQL layer.

### 2. Create a helper to build per-table `OrderBy` input types in `graphql.py`

Add a function `create_order_by_input(table: Table) -> type` that:

1. Iterates over all `table.columns`
2. Creates a dict of `{column_name: (SortDirection | None, strawberry.UNSET)}` annotations
3. Uses `strawberry.input` + `type()` (dynamic class creation) to produce a Strawberry input class named `{PascalTableName}OrderBy`

This follows the same dynamic type creation pattern already used for `gql_type` on line 272:
```python
gql_type = mapper.type(orm_class)(type(table.name, (object,), {}))
```

### 3. Create a helper to apply order_by to a SQLAlchemy statement in `graphql.py`

Add a function `apply_order_by(statement, orm_class, order_by_input)` that:

1. Iterates over the fields of the order_by input instance
2. For each field that is not `UNSET`/`None`, gets the corresponding SQLAlchemy column via `getattr(orm_class, field_name)`
3. Maps the `SortDirection` value to the correct SQLAlchemy clause:
   - `ASC` → `column.asc()`
   - `ASC_NULLS_FIRST` → `column.asc().nulls_first()`
   - `ASC_NULLS_LAST` → `column.asc().nulls_last()`
   - `DESC` → `column.desc()`
   - `DESC_NULLS_FIRST` → `column.desc().nulls_first()`
   - `DESC_NULLS_LAST` → `column.desc().nulls_last()`
4. Chains `.order_by()` onto the statement

### 4. Update `create_resolver` to accept and use the order_by input type

- Add `order_by_type` parameter to `create_resolver()`
- Change `list_resolver` signature: replace `order_by: str | None = None` with `order_by: order_by_type | None = None`
- Before query execution, call `apply_order_by()` if `order_by` is not `None`

### 5. Update `build()` to create and pass order_by types

In the `build()` loop, after creating `gql_type`:

1. Call `create_order_by_input(table)` to get the per-table input type
2. Pass it to `create_resolver(table.name, gql_type, ResolverType.LIST, order_by_type=...)`

### 6. Add `StrawberryArgument` for order_by on list resolvers

Similar to how pk arguments are dynamically injected (lines 290-295), inject the `order_by` argument onto the list resolver's `base_resolver.arguments` so Strawberry registers it in the schema with the correct per-table input type.

## Notes

- The `order_by` input is a single object (not a list). Multiple columns can be specified as separate fields on the same object. SQLAlchemy applies them in iteration order (Python 3.7+ dicts are insertion-ordered).
- Fields default to `UNSET` (not `None`) so omitted columns produce no ordering clause.
- No changes needed to `persistence.py`, `rest.py`, or `main.py`.
